用户权限管理模块功能笔记

它采用的是标准的基于角色的访问控制（RBAC）模型，并结合 JWT 来实现安全的身份认证。下面是具体机制和过程的分步说明：

  1. 核心组件 (构成基础):

    • 用户 (SysUser): 代表与系统交互的个体用户。每个用户拥有唯一的凭证（如用户名/密码）和相关的个人信息。通过 SysUserController 和 SysUserService 进行管理。
    • 角色 (SysRole): 代表系统内的一组权限或一个工作职能（例如，“管理员”、“学生”）。角色是用户和权限之间的中间层。通过 SysRoleController 和 SysRoleService 进行管理。
    • 菜单 (SysMenu): 代表系统的功能，通常在用户界面中表现为菜单项。这些是实际被授予权限的资源或操作（例如，“查看用户列表”、“编辑阅览室”）。通过 SysMenuController 和 SysMenuService 进行管理。

  2. 关系 (连接组件):

    • 用户-角色关联 (SysUserRole): 定义了哪个用户拥有哪些角色。一个用户可以被分配一个或多个角色。这种关系通常通过一个连接表 (sys_user_role) 和相应的服务 (SysUserRoleService) 来管理，允许灵活分配。例如，某个特定用户可能同时被分配“学生”和“图书馆助理”的角色。
    • 角色-菜单关联 (RoleMenu): 定义了哪个角色有权限访问哪些菜单（即权限）。一个角色可以被授予访问多个菜单项的权限。这也通过一个连接表 (sys_role_menu) 及其服务 (RoleMenuService) 来管理。例如，“管理员”角色很可能关联所有菜单，而“学生”角色可能只被允许访问与座位预订和查看通知相关的菜单。

  3. 身份认证过程 (登录并获取“钥匙”):

    • 登录请求: 用户通过发送他们的凭证（例如，用户名和密码）到一个特定接口（很可能由 SysUserController 处理）来发起登录。
    • 凭证验证: 系统接收到凭证。CustomerUserDetailService 被调用。它尝试通过用户名从数据库加载用户 (SysUserService.loadUser)。如果用户存在且密码匹配（经过哈希比对后），则身份认证成功。
    • JWT 生成: 认证成功后，系统使用 JwtUtils.generateToken 创建一个 JSON Web Token (JWT)。这个令牌是一个紧凑、自包含的字符串，包含了关于用户的信息（声明，如用户ID、用户名）和元数据（如过期时间）。它使用一个密钥（来自配置文件的 jwt.secret）进行签名，以防篡改。
    • 返回令牌: 生成的 JWT 被发送回用户的客户端（例如，网页浏览器）。

  4. 授权过程 (使用“钥匙”并检查权限):

    • 后续请求: 对于之后访问受保护资源（除了登录、图片等之外的大多数API）的任何请求，客户端必须在请求中包含接收到的 JWT，通常放在 Authorization 请求头中（例如 Authorization: Bearer <token>）或作为请求参数。
    • 令牌过滤 (CheckTokenFilter): 这个过滤器在请求到达控制器之前拦截所有传入请求。
      • 它检查请求的 URL 是否在 ignoreUrl 列表（如登录、图片、某些微信API）中。如果是，过滤器不再做进一步处理，请求继续进行。
      • 如果 URL 不在忽略列表中，过滤器从请求头或参数中提取 JWT。
      • 它使用 JwtUtils.verify 验证令牌（检查签名和过期时间）。如果无效，抛出异常，访问被拒绝（由 LoginFailureHandler 处理）。
      • 如果有效，它使用 JwtUtils.jwtDecode 解码令牌以提取声明，特别是 username。
    • 加载用户详情和权限 (CustomerUserDetailService):
      • 过滤器使用提取到的 username 调用 customerUserDetailService.loadUserByUsername(username)。
      • 这个服务方法获取完整的 SysUser 对象。关键在于，它接着确定用户的权限：
        • 它检查用户是否是管理员 (isAdmin == "1")。如果是，它获取所有菜单 (sysMenuService.list())。
        • 如果不是管理员，它仅获取通过用户角色分配给该用户的菜单 (sysMenuService.getMenuByUserId(user.getUserId()))。这个查询很可能连接了 sys_user, sys_user_role, sys_role_menu, 和 sys_menu 这几个表。
      • 将这些允许访问的菜单的编码/标识符（例如 item.getCode()）收集起来，并转换为 GrantedAuthority 对象（代表权限）。
      • 返回填充完整的 SysUser 对象（它实现了 UserDetails 接口），现在包含了用户的权限（authorities）。
    • 设置安全上下文: CheckTokenFilter 获取返回的 UserDetails，并创建一个 Authentication 对象（具体是 UsernamePasswordAuthenticationToken）。这个对象代表了已认证的用户及其被授予的权限，并被放入 SecurityContextHolder 中。这使得用户的身份和权限在整个请求处理周期中都可用。
    • 访问控制决策: 最后，Spring Security 使用 SecurityContextHolder 中的信息。当请求到达一个带有安全约束注解（例如 @PreAuthorize("hasAuthority('sys:user:list')")）或在 SpringSecurityConfig 中配置了访问规则的控制器方法时，Spring Security 会检查 Authentication 对象是否包含所需的权限。如果包含，访问被允许；否则，访问被拒绝（由 CustomAccessDeineHandler 处理）。前端界面显示的菜单列表通常也是在登录后根据这些授予的权限来获取的。

简单来说，系统首先验证用户是谁（通过 JWT 进行身份认证），然后确定他们能做什么（通过 RBAC 角色及在令牌验证期间动态加载的关联菜单权限进行授权）。这种结合提供了一种安全且灵活的方式来管理整个应用程序的访问控制。
